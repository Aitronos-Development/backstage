# Freddy Backend in Backstage

**Status:** Draft
**Author:** Platform Engineering Team
**Date:** February 2026

---

## The Problem

Freddy Backend is a FastAPI service with ~30 API route groups, four infrastructure dependencies (PostgreSQL, Redis, Qdrant, limit enforcement), and a growing team working on it. Today, a developer who wants to understand what APIs Freddy exposes has to open Swagger UI directly. To check if the service is healthy, they have to curl an endpoint or check logs. There's no single place that says: "here's Freddy, here's what it does, and here's whether it's working."

## The Goal

Give Freddy Backend a **home page** inside Backstage where a developer can:

1. **Find the service** — search for "Freddy Backend" in the Backstage catalog and land on its dedicated page
2. **Browse all API routes** — see every parent route group (`/v1/health`, `/v1/auth`, `/v1/assistants`, `/v1/threads`, etc.) rendered as interactive API documentation, directly inside Backstage
3. **See live health status** — a widget on the service page that shows whether Freddy and its components (database, Redis, Qdrant, limit enforcement) are healthy, degraded, or down — in real time, with green/yellow/red indicators

## What This Is Not

This is a **minimum first step**. We are intentionally keeping scope small:

- No Prometheus or metrics collection
- No SonarQube or code quality analysis
- No TechDocs or rendered documentation
- No CI/CD pipeline visibility in Backstage
- No alerting or uptime history

These can all be added later as separate iterations. For now, the only question we're answering is: **"What is Freddy, what does it expose, and is it working right now?"**

## What Already Exists

Freddy Backend is not starting from zero. The following are already in place:

- **Health endpoints** — `/v1/health/` (liveness) and `/v1/health/details` (readiness with per-component status)
- **OpenAPI spec** — auto-generated by FastAPI, served at `/openapi.json`, covering all ~30 route groups
- **CI/CD pipelines** — 13 GitHub Actions workflows for testing, deployment across develop/staging/production
- **Docker-based infrastructure** — PostgreSQL, Redis, and Qdrant all running as containers

What does **not** exist yet:

- A Backstage instance
- A catalog entry for Freddy Backend
- A health check widget for Backstage

## What the Developer Sees

A developer clicks on a service in the catalog. The service entity page has **three tabs**: **Overview**, **API**, and **TechDocs**.

### Overview Tab

- **Service name** displayed prominently
- **Health & Liveness card** — rendered inline on the page, showing real-time status directly on the card (not a link to another page):

| Component           | Status  |
| ------------------- | ------- |
| Overall             | Healthy |
| Database            | Healthy |
| Redis Cache         | Healthy |
| Limit Enforcement   | Healthy |
| Qdrant Vector Store | Healthy |

Each row is green, yellow, or red. The card auto-refreshes.

### API Tab

The API tab has two sub-headers:

- **Produced APIs** — APIs this service provides
- **Consumed APIs** — APIs this service consumes

Under each sub-header, **parent route groups** are listed (e.g., `/v1/health`, `/v1/auth`, `/v1/assistants`). Each parent route is **collapsible** — clicking it expands to show the individual endpoints connected to that route.

### TechDocs Tab

Rendered documentation for the service.

## Scope Summary

| In scope                                            | Out of scope (future)      |
| --------------------------------------------------- | -------------------------- |
| New Backstage instance                              | Prometheus / Grafana       |
| Freddy registered in catalog                        | SonarQube integration      |
| API docs rendered from OpenAPI spec                 | GitHub Actions plugin      |
| Live health status widget (inline, not a redirect)  | Alerting or uptime history |
| Three tabs: Overview, API, TechDocs                 |                            |
| API tab with Produced/Consumed sub-headers          |                            |
| Collapsible parent route groups under each API type |                            |
| TechDocs tab on service page                        |                            |

## Success Criteria

- A developer can find "Freddy Backend" in the Backstage catalog
- Clicking on it shows a page with three tabs: Overview, API, TechDocs
- Overview tab shows the service name and an inline health card with liveness status for all components (no redirect)
- API tab shows Produced APIs and Consumed APIs as sub-headers
- Under each sub-header, parent route groups are listed and collapsible to reveal individual endpoints
- TechDocs tab renders documentation
- No external tooling beyond Backstage and the running Freddy service is required
